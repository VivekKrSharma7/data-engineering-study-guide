## Introduction

The payment waterfall is the core mechanism of every structured finance transaction. It defines the precise order in which cash flows generated by the mortgage pool are distributed to bondholders, servicers, trustees, and other deal parties. In private-label RMBS, waterfalls can be extraordinarily complex — involving dozens of priority steps, conditional logic based on triggers, and different rules for interest versus principal. For data engineers, implementing waterfall logic accurately is one of the most critical and technically challenging tasks in MBS analytics.

---

## Key Concepts

### 1. Payment Waterfall Concept

A waterfall is an ordered set of rules specifying how available funds are distributed in each payment period:

```
Waterfall Analogy:

     Available Funds
          |
          v
    +-----------+
    | Priority 1 |  <-- Filled first (e.g., trustee fees)
    +-----------+
          |  (overflow)
          v
    +-----------+
    | Priority 2 |  <-- Filled next (e.g., servicing fees)
    +-----------+
          |  (overflow)
          v
    +-----------+
    | Priority 3 |  <-- Then next (e.g., senior interest)
    +-----------+
          |  (overflow)
          v
       ... and so on through all priority levels ...
          |
          v
    +-----------+
    | Residual   |  <-- Whatever remains flows to residual holder
    +-----------+
```

Each priority level must be fully satisfied before any funds flow to the next level. If available funds are exhausted at any point, lower-priority items receive nothing.

### 2. Priority of Payments — Typical Non-Agency RMBS

A typical post-crisis non-agency RMBS waterfall has separate interest and principal waterfalls:

#### Interest Waterfall (Simplified)

```
Available Interest Funds:
  = Interest collected from performing loans
  + Net recoveries on defaulted loans (interest portion)
  - Servicer advances recouped

Priority of Interest Payments:

1.  Trustee fees and expenses
2.  Servicing fee (master servicer, primary servicer, special servicer)
3.  Senior tranche (Class A) interest
4.  Senior tranche (Class A) interest shortfall carryforward
5.  Class M-1 interest
6.  Class M-1 interest shortfall carryforward
7.  Class M-2 interest
8.  Class M-2 interest shortfall carryforward
9.  Class B-1 interest
10. Class B-1 interest shortfall carryforward
11. Class B-2 interest
12. Class B-2 interest shortfall carryforward
13. Residual (remaining interest funds)
```

#### Principal Waterfall (Simplified — Sequential Mode)

```
Available Principal Funds:
  = Scheduled principal payments
  + Prepayments (curtailments + payoffs)
  + Net recoveries on defaulted loans (principal portion)
  + Excess spread directed to principal (if OC below target)

Priority of Principal Payments (Sequential):

1.  Class A principal (until balance = $0)
2.  Class M-1 principal (until balance = $0)
3.  Class M-2 principal (until balance = $0)
4.  Class B-1 principal (until balance = $0)
5.  Class B-2 principal (until balance = $0)
6.  Over-collateralization release (if above target and triggers passing)
```

#### Principal Waterfall (Pro-Rata Mode — After Step-Down, Triggers Passing)

```
Priority of Principal Payments (Pro-Rata):

1.  Distribute principal pro-rata to all tranches based on current balance
    (subject to shifting interest percentages for subordinate classes)
2.  Any excess principal to build OC if below target
3.  OC release if above target
```

### 3. Interest Waterfall — Detailed Mechanics

#### Available Funds Cap

The available funds cap limits the interest that can be paid on any tranche to the actual interest collected from the pool:

```
Available Funds Cap:

If collected interest is insufficient to pay all bond interest:
  Bond Interest Due = Min(Stated Coupon Rate × Balance, Available Interest Funds)

This creates an "available funds cap shortfall" — interest that
cannot be paid because the pool didn't generate enough cash.

Key distinction:
  - Available Funds Cap Shortfall: NOT a credit loss;
    typically forgiven or written off
  - Interest Shortfall due to losses: Results from defaulted
    loans reducing the pool; may be recoverable
```

#### Net WAC Cap

Many RMBS tranches have a Net WAC (weighted average coupon) cap that limits the tranche coupon to the pool's net WAC:

```
Net WAC Cap:

Tranche Coupon = Min(Stated Coupon, Net WAC - Senior Fees)

Example:
  Pool WAC:          5.75%
  Servicing Fee:     0.25%
  Trustee Fee:       0.02%
  Net WAC:           5.48%

  Class A-1 Stated Coupon: 5.50%
  Class A-1 Actual Coupon: Min(5.50%, 5.48%) = 5.48%

  Net WAC Cap Shortfall: (5.50% - 5.48%) × A-1 Balance / 12

Net WAC Cap Carryover:
  Some deals allow the shortfall to be recovered in future periods
  if Net WAC subsequently exceeds the stated coupon.
```

### 4. Loss Allocation — Reverse Sequential

Losses are allocated in reverse order of seniority (bottom-up):

```
Loss Allocation Example:

Realized Loss this Period: $2,000,000

Step 1: Apply to OC
  OC Balance: $3,000,000
  Loss Applied: $2,000,000
  Remaining OC: $1,000,000
  Remaining Loss: $0

If losses exceed OC:

Realized Loss this Period: $5,000,000

Step 1: Apply to OC
  OC Balance: $3,000,000
  Loss Applied: $3,000,000
  Remaining OC: $0
  Remaining Loss: $2,000,000

Step 2: Apply to B-2 (first-loss tranche)
  B-2 Balance: $10,000,000
  Loss Applied: $2,000,000
  Remaining B-2: $8,000,000
  Remaining Loss: $0

If losses continue:

Step 3: Apply to B-1 (next subordinate)
Step 4: Apply to M-2
Step 5: Apply to M-1
Step 6: Apply to Class A (senior takes losses last)
```

**Writedown mechanics:**
- When a loss is allocated to a tranche, its outstanding principal balance is written down.
- Written-down amounts may be reimbursable from future excess spread (if the deal allows).
- Writedown reduces future interest payments (calculated on lower balance).
- Some deals distinguish between "realized losses" (permanent, from liquidation) and "applied loss amounts" (may include collateral deficiency amounts).

### 5. Trigger Mechanics in the Waterfall

Triggers determine which waterfall branch is executed:

```
Waterfall Decision Tree:

               +-----------------------+
               | Step-Down Date Passed? |
               +-----------------------+
                /                    \
              No                     Yes
              |                       |
       Sequential Pay          +------------------+
                              | All Triggers Pass? |
                              +------------------+
                               /              \
                             No               Yes
                             |                 |
                      Sequential Pay     Pro-Rata Pay
                      (protective)    (with shifting interest)
```

**Trigger implementation for data engineers:**

```python
# Pseudocode for trigger evaluation

def evaluate_triggers(deal_state, period):
    """Evaluate all triggers for a given period."""

    # Cumulative Loss Trigger
    cum_loss_pct = deal_state.cumulative_losses / deal_state.original_balance
    cum_loss_threshold = get_cum_loss_threshold(period)
    cum_loss_trigger_pass = cum_loss_pct < cum_loss_threshold

    # Delinquency Trigger
    dq_balance = (deal_state.dq_60_plus + deal_state.fc_balance
                  + deal_state.reo_balance)
    dq_pct = dq_balance / deal_state.current_balance
    dq_threshold = deal_state.initial_sub_pct * 0.50  # 50% of initial sub
    dq_trigger_pass = dq_pct < dq_threshold

    # CE Trigger
    current_ce_pct = deal_state.subordinate_balance / deal_state.current_balance
    ce_target = deal_state.initial_ce_pct * 2.0  # Must double initial CE%
    ce_trigger_pass = current_ce_pct >= ce_target

    # Step-down eligible only if ALL triggers pass
    step_down_eligible = (period >= deal_state.step_down_month
                         and cum_loss_trigger_pass
                         and dq_trigger_pass
                         and ce_trigger_pass)

    return step_down_eligible
```

### 6. Clean-Up Calls

A clean-up call allows the deal to be terminated early when the remaining pool balance falls below a threshold:

```
Clean-Up Call:

Typical Threshold: 10% of original pool balance (sometimes 20%)

When Triggered:
  - Servicer or residual holder has the OPTION (not obligation) to call
  - Purchases remaining loans at par (or specified price)
  - Proceeds used to retire all outstanding bonds
  - Remaining value accrues to the caller

Why it Exists:
  - Small pools are expensive to service relative to revenue
  - Trustee and administrative fees become disproportionate
  - Allows efficient deal termination

Data Engineering Consideration:
  - Must track pool factor (current balance / original balance)
  - Alert when approaching clean-up threshold
  - Model cash flows both with and without clean-up exercise
```

### 7. Modeling Waterfalls

#### Waterfall Modeling Steps

```
Step 1: Project Collateral Cash Flows
  Input: Loan-level data + assumptions (CPR, CDR, severity, lag)
  Output: Monthly schedule of:
    - Scheduled interest
    - Scheduled principal
    - Prepayments
    - Defaults
    - Recoveries (with timing lag)
    - Losses

Step 2: Calculate Available Funds
  Available Interest = Collected Interest + Recovery Interest - Advances Recouped
  Available Principal = Scheduled Prin + Prepays + Recovery Prin + Excess Spread (if applicable)

Step 3: Determine Waterfall Mode
  Evaluate step-down date and trigger conditions
  Select appropriate waterfall branch (sequential or pro-rata)

Step 4: Distribute Interest
  Follow interest priority of payments
  Apply available funds cap and net WAC cap
  Track shortfalls and carryforwards

Step 5: Allocate Losses
  Apply losses reverse-sequentially (bottom-up)
  Write down tranche balances
  Check for excess spread reimbursement

Step 6: Distribute Principal
  Follow principal priority of payments
  Apply shifting interest percentages if applicable
  Check OC target and build/release

Step 7: Calculate Period Outputs
  - Tranche balances (beginning and ending)
  - Cash flows paid (interest and principal)
  - Shortfalls and writedowns
  - Trigger status
  - CE levels
```

#### Key Data Fields for Waterfall Modeling

| Category | Data Fields |
|----------|-------------|
| **Loan-Level** | Current balance, note rate, maturity date, payment status, months delinquent, modification flag, modification terms |
| **Pool-Level** | Aggregate balance, WAC, WAM, WALA, delinquency buckets (30/60/90/120+/FC/REO), prepayment speed, default rate, severity, recovery lag |
| **Deal-Level** | Tranche balances, coupon rates (fixed/float/formula), credit enhancement, trigger thresholds, step-down dates, shifting interest schedule, OC target/floor |
| **Period Outputs** | Available funds, distributions by tranche, losses allocated, writedowns, reimbursements, trigger status, ending balances |

### 9. Advanced Waterfall Concepts

#### Servicer Advancing

Many deals require servicers to advance principal and interest on delinquent loans:

```
Servicer Advancing:

Month N: Borrower misses payment
  Servicer advances P&I to the trust (if deemed recoverable)
  Trust distributes normally to bondholders

Month N+6: Loan liquidated
  Recovery proceeds first reimburse servicer advances
  Remaining recovery applied as collateral cash flow
  Shortfall between advance and recovery = realized loss

Impact on Waterfall:
  - Advances smooth cash flows for bondholders
  - But advance recovery has priority over bondholder distributions
  - "Stop advance" determination (non-recoverability) creates
    immediate shortfall in available funds
```

#### Modified Loans in the Waterfall

```
Modification Impact:

Pre-Modification:
  Loan Balance: $300,000
  Rate: 6.50%
  Monthly P&I: $1,896

Post-Modification (rate reduction):
  Loan Balance: $300,000
  Rate: 4.00%
  Monthly P&I: $1,432

Impact on Waterfall:
  - Reduced interest income: $464/month per modified loan
  - Pool WAC decreases
  - Excess spread shrinks
  - Net WAC cap may bind on bond coupons
  - Principal forgiveness (if any) is an immediate realized loss

Data Fields to Track:
  - Pre-mod rate, post-mod rate
  - Modification type (rate reduction, term extension,
    principal forbearance, principal forgiveness)
  - Step-rate schedules (temporary modifications)
  - Trial modification period
```

#### Realized Loss vs. Writedown

```
Important Distinction:

Realized Loss:
  - Permanent loss from loan liquidation
  - Loan Balance - Liquidation Proceeds - Costs = Realized Loss
  - Allocated to tranches bottom-up
  - May be partially reimbursable from future excess spread

Applied Loss Amount / Writedown:
  - May include "collateral deficiency amount" for deeply
    delinquent loans still in the pool
  - Tranche balance written down to reflect expected loss
  - Can be reversed if loan subsequently cures or
    liquidates at better-than-expected recovery

Data Engineering Note:
  Must carefully distinguish these in the data model;
  different deals use different terminology and mechanics.
```

## Common Interview Questions & Answers

### Q1: Explain the payment waterfall concept in private-label RMBS.

**Answer**: A payment waterfall is the contractually defined priority of payments that governs how cash flows from the mortgage pool are distributed each month. It is specified in the PSA and operates like a cascading priority scheme — each level must be fully satisfied before funds flow to the next. There are typically separate waterfalls for interest and principal. The interest waterfall pays fees first (trustee, servicer), then interest to tranches in order of seniority, with any excess becoming available for loss absorption or OC building. The principal waterfall distributes principal either sequentially (most senior first) or pro-rata (proportionally), depending on the deal's trigger status and whether the step-down date has been reached. Losses are allocated in reverse — bottom-up through the capital structure. The waterfall's conditional logic (triggers, shifting interest, OC targets) makes it one of the most complex components to implement in a data engineering context.

### Q2: What is the difference between sequential and pro-rata principal distribution, and when does each apply?

**Answer**: In sequential distribution, all principal goes to the most senior outstanding tranche until it is fully retired, then the next tranche, and so on. This rapidly de-levers senior tranches and increases their credit enhancement. In pro-rata distribution, principal is shared proportionally among all tranches based on their current balances, per the shifting interest schedule. Sequential applies during the initial lockout period (first 48–60 months) and any time performance triggers are failing. Pro-rata applies after the step-down date only if all triggers are passing. The key risk of pro-rata is that it allows subordinate tranches to pay down, which maintains but does not increase CE percentages. This is why triggers exist — to revert to sequential (protective mode) if collateral performance deteriorates, ensuring senior bondholders are protected.

### Q3: How are losses allocated in an RMBS structure?

**Answer**: Losses are allocated in reverse sequential order — from the bottom of the capital structure upward. First, realized losses reduce over-collateralization. Once OC is exhausted, losses write down the first-loss tranche (lowest-rated or unrated class). When that tranche is fully written down, losses hit the next tranche up, continuing through mezzanine classes and ultimately to the senior tranche if losses are severe enough. Excess spread provides an additional buffer — current-period excess spread can absorb losses before they reach OC or bond tranches. Some deals also allow previously written-down amounts to be reimbursed from future excess spread. As a data engineer, accurately tracking loss allocation requires maintaining the state of each tranche balance, OC level, and cumulative excess spread usage across all periods.

### Q4: What is the Net WAC cap and the available funds cap?

**Answer**: The Net WAC cap limits a tranche's coupon rate to the pool's net weighted average coupon minus senior fees. If the pool's net WAC falls below a tranche's stated coupon (due to modifications, prepayment of high-rate loans, or ARM resets), the tranche receives less interest than its stated rate. The shortfall may or may not carry forward depending on deal terms. The available funds cap limits total interest paid to what was actually collected from borrowers — if collections are insufficient (due to delinquencies or defaults), bond interest is capped at available funds. Available funds cap shortfalls are typically non-recoverable and represent a permanent interest loss to bondholders, whereas Net WAC cap shortfalls may have carryover provisions. Both caps are important for data engineers to model accurately because they affect tranche yields, especially in stress scenarios.

### Q5: How does Intex work and how do data engineers integrate with it?

**Answer**: Intex is the industry-standard vendor that maintains a library of virtually all structured finance deals with their waterfall logic coded from the PSA. Data engineers interact with Intex primarily through its API (INTEXcalc). The workflow is: (1) specify a deal by CUSIP or deal name; (2) provide collateral assumptions — prepayment speed (CPR), default rate (CDR), loss severity, recovery lag, and interest rate scenario; (3) Intex runs the assumptions through its coded waterfall and returns tranche-level monthly cash flows — principal, interest, losses, factors, and derived analytics like yield and WAL. Data engineers build pipelines that programmatically submit assumption sets for portfolios of deals, capture the output, store it in data warehouses, and feed it into downstream analytics (OAS calculations, scenario analysis, risk reporting). Intex is considered the authoritative source for waterfall calculations, and firms typically validate their in-house models against Intex output.

### Q6: What are the most challenging aspects of implementing waterfall logic from a data engineering perspective?

**Answer**: The top challenges are: (1) **Complexity and variability** — every deal has a unique waterfall with different priorities, triggers, and rules; there is no universal template; (2) **Conditional logic** — triggers create branching paths in the waterfall that depend on real-time performance data; (3) **State management** — waterfalls are stateful; each period's ending state (balances, cumulative losses, shortfall carryforwards) becomes the next period's starting state; (4) **Edge cases** — rare scenarios like available funds cap binding, Net WAC cap shortfalls, servicer stop-advance decisions, or deal amendments require handling; (5) **Data quality** — trustee reports may have inconsistencies, restatements, or format changes that must be reconciled; (6) **Validation** — results must be validated against Intex, trustee reports, and rating agency models; (7) **Scale** — running waterfall models across thousands of deals under multiple scenarios requires efficient computation and storage.

## Tips for Interview Success

1. **Walk through a waterfall step by step**: If asked about waterfalls, methodically walk through the interest waterfall, then principal, then loss allocation. Show that you understand the complete flow.

2. **Know the state variables**: Demonstrate awareness that waterfalls are stateful — you need current balances, cumulative losses, shortfall carryforwards, trigger status, and OC levels from the prior period.

3. **Emphasize the PSA**: The PSA is the source of truth. Any waterfall model must be traceable to PSA provisions. Show you understand this is not a generic formula but a deal-specific legal document.

4. **Intex familiarity**: Mentioning Intex and understanding how it fits into the technology stack shows practical industry knowledge that book learning alone does not provide.

5. **Think about edge cases**: Interviewers appreciate when candidates proactively mention edge cases — what happens when excess spread goes negative, when a trigger cures and then fails again, or when a clean-up call is exercised.

6. **Connect to data pipelines**: Frame your answer around building data systems — ingesting trustee reports, running waterfall models, storing results, and feeding downstream analytics. This shows you think like an engineer, not just an analyst.
