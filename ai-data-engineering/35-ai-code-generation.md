## Overview

AI code generation tools have fundamentally changed the productivity floor for data engineers. For a senior engineer with 16+ years of SQL Server and Snowflake experience, these tools are most powerful as accelerators for boilerplate-heavy work: T-SQL stored procedures, Snowpark transformations, dbt models, Airflow DAGs, and SSIS-to-Python migrations. Understanding how these models work — and where they fail — separates engineers who use them effectively from those who ship AI-hallucinated SQL to production.

| Concept | Description |
|---|---|
| Code LLMs | Language models pre-trained on code corpora (GitHub, GitLab, etc.) |

| Fine-tuning | Training a code model on your company's specific SQL patterns |

## How Code Generation Models Work

### Training Data and Architecture

Code LLMs (CodeLlama, StarCoder2, DeepSeek-Coder) are pre-trained on:
- Public GitHub repositories (filtered for quality/license)
- Documentation, Stack Overflow, technical books
- Synthetic code generated by larger models

Key architectural features relevant to data engineers:
- **Fill-in-the-Middle (FIM)**: Model can complete code given prefix AND suffix context — used for Copilot's inline completions
- **Long context**: Claude 3.5 Sonnet supports 200K tokens — enough for a full stored procedure library
- **Code-specific tokenization**: Code tokens (identifiers, operators) tokenized more efficiently than natural language models

### Why SQL Is Harder Than Python for AI

```
Challenge                      | Impact
-------------------------------|------------------------------------------
Schema dependency              | Model must know table/column names
Join semantics                 | Implicit vs explicit, ANSI vs vendor
Vendor dialects                | T-SQL ≠ Snowflake SQL ≠ ANSI SQL
Window function nuances        | ROWS vs RANGE, frame defaults differ
Execution plan awareness       | Model can't reason about indexes/stats
Implicit type coercion         | Platform-specific behavior
CTEs vs subqueries tradeoffs   | Model chooses arbitrarily
```

## AI for T-SQL: Practical Examples

### Generating DMV Queries for Performance Analysis

```sql
-- AI Prompt: "Generate a T-SQL query to find the top 20 most expensive queries
-- by total CPU time from the plan cache, including execution count,
-- average duration, and the query text truncated to 200 chars"

-- Claude generates:
SELECT TOP 20
    qs.total_worker_time / 1000                     AS total_cpu_ms,
    qs.execution_count,
    qs.total_worker_time / qs.execution_count / 1000 AS avg_cpu_ms,
    qs.total_elapsed_time / qs.execution_count / 1000 AS avg_duration_ms,
    SUBSTRING(qt.text, (qs.statement_start_offset / 2) + 1,
        ((CASE qs.statement_end_offset
            WHEN -1 THEN DATALENGTH(qt.text)
            ELSE qs.statement_end_offset
          END - qs.statement_start_offset) / 2) + 1) AS query_text
FROM sys.dm_exec_query_stats AS qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) AS qt
ORDER BY qs.total_worker_time DESC;
```

### AI-Generated Index Recommendation Analysis

```sql
-- Prompt: "Query missing index DMVs for the loan_performance table,
-- ranked by estimated improvement. Show the suggested index DDL."

SELECT
    migs.avg_total_user_cost * migs.avg_user_impact * (migs.user_seeks + migs.user_scans) AS improvement_measure,
    mid.statement                          AS table_name,
    mid.equality_columns,
    mid.inequality_columns,
    mid.included_columns,
    CONCAT(
        'CREATE INDEX IX_', REPLACE(mid.statement, '[dbo].', ''), '_',
        REPLACE(ISNULL(mid.equality_columns, ''), ', ', '_'),
        ' ON ', mid.statement,
        ' (', ISNULL(mid.equality_columns, ''), ')',
        CASE WHEN mid.included_columns IS NOT NULL
             THEN CONCAT(' INCLUDE (', mid.included_columns, ')')
             ELSE '' END
    ) AS suggested_index_ddl
FROM sys.dm_db_missing_index_groups AS mig
JOIN sys.dm_db_missing_index_group_stats AS migs ON mig.index_group_handle = migs.group_handle
JOIN sys.dm_db_missing_index_details AS mid ON mig.index_handle = mid.index_handle
WHERE mid.statement LIKE '%loan_performance%'
ORDER BY improvement_measure DESC;
```

## AI for Snowflake: Snowpark and dbt

### Generating Snowpark Code

```python
# Prompt to Claude:
# "Write a Snowpark Python UDF that computes weighted average coupon (WAC)
# for a pool of mortgages. Input: pool_id, upb array, note_rate array.
# Register it in Snowflake and show the SQL to call it."

from snowflake.snowpark import Session
from snowflake.snowpark.functions import udf
from snowflake.snowpark.types import FloatType, ArrayType, StringType
import numpy as np

session = Session.builder.configs(connection_params).create()

@udf(name='compute_wac',
     input_types=[ArrayType(FloatType()), ArrayType(FloatType())],
     return_type=FloatType(),
     replace=True,
     session=session)
def compute_wac(upb_list: list, rate_list: list) -> float:
    """Weighted Average Coupon: sum(UPB * rate) / sum(UPB)"""
    if not upb_list or not rate_list or sum(upb_list) == 0:
        return None
    total_upb = sum(upb_list)
    weighted_rate = sum(u * r for u, r in zip(upb_list, rate_list))
    return weighted_rate / total_upb
```

```sql
-- SQL call generated by AI:
SELECT
    pool_id,
    compute_wac(
        ARRAY_AGG(current_upb) WITHIN GROUP (ORDER BY loan_id),
        ARRAY_AGG(note_rate) WITHIN GROUP (ORDER BY loan_id)
    ) AS wac
FROM loan_performance
WHERE report_month = '2026-02-01'
GROUP BY pool_id;
```

### AI-Generated dbt Model

```sql
-- Prompt: "Generate a dbt incremental model for loan-level CPR calculation.
-- Source: {{ ref('stg_loan_performance') }}. Incremental on report_month.
-- Handle first-month loans (no prior period) gracefully."

{{
  config(
    materialized='incremental',
    unique_key=['loan_id', 'report_month'],
    incremental_strategy='merge',
    cluster_by=['report_month']
  )
}}

WITH performance AS (
    SELECT
        loan_id,
        report_month,
        current_upb,
        scheduled_principal,
        LAG(current_upb) OVER (PARTITION BY loan_id ORDER BY report_month) AS prior_upb
    FROM {{ ref('stg_loan_performance') }}
    {% if is_incremental() %}
        WHERE report_month >= (SELECT MAX(report_month) FROM {{ this }})
    {% endif %}
),

smm_calc AS (
    SELECT
        loan_id,
        report_month,
        current_upb,
        prior_upb,
        CASE
            WHEN prior_upb IS NULL OR prior_upb = 0 THEN NULL
            ELSE (prior_upb - current_upb - scheduled_principal) / prior_upb
        END AS smm
    FROM performance
)

SELECT
    loan_id,
    report_month,
    smm,
    CASE
        WHEN smm IS NULL THEN NULL
        WHEN smm >= 1    THEN 1.0
        WHEN smm <= -1   THEN 0.0
        ELSE 1 - POWER(1 - smm, 12)
    END AS cpr,
    CURRENT_TIMESTAMP() AS dbt_updated_at
FROM smm_calc
```

## Building a SQL Generator with the Claude API

```python
import anthropic
import json
from typing import Optional

class MortgageSQLGenerator:
    """
    Custom SQL generator using Claude API for mortgage data engineering.
    Provides schema context automatically from a schema registry.
    """

    def __init__(self, api_key: str, schema_registry_path: str = 'schemas.json'):
        self.client = anthropic.Anthropic(api_key=api_key)
        with open(schema_registry_path) as f:
            self.schema_registry = json.load(f)

    def _build_schema_context(self, table_names: list[str]) -> str:
        """Extract relevant table schemas for prompt context."""
        schema_lines = []
        for table in table_names:
            if table in self.schema_registry:
                schema = self.schema_registry[table]
                cols = ', '.join(
                    f"{c['name']} {c['type']}" for c in schema['columns']
                )
                schema_lines.append(f"[{table}]: {cols}")
        return '\n'.join(schema_lines)

    def generate_sql(
        self,
        request: str,
        tables: list[str],
        dialect: str = 'snowflake',
        validate: bool = True
    ) -> dict:
        """Generate SQL from natural language request with schema context."""

        schema_context = self._build_schema_context(tables)

        system_prompt = f"""You are an expert {dialect.upper()} SQL developer
specializing in secondary mortgage market analytics.

Database dialect: {dialect}
Coding standards:
- snake_case column aliases
- CTEs preferred over subqueries
- All monetary values in DECIMAL(18,2)
- Include comments for complex logic
- Handle NULL and division-by-zero cases explicitly

Available schema:
{schema_context}

Return ONLY the SQL query, no explanation. If the request is ambiguous,
add a comment at the top explaining your assumption."""

        response = self.client.messages.create(
            model='claude-opus-4-5',
            max_tokens=2048,
            system=system_prompt,
            messages=[{'role': 'user', 'content': request}]
        )

        sql = response.content[0].text.strip()

        result = {
            'sql': sql,
            'tables_used': tables,
            'dialect': dialect,
            'input_tokens': response.usage.input_tokens,
            'output_tokens': response.usage.output_tokens
        }

        if validate:
            result['validation'] = self._basic_validate(sql, dialect)

        return result

    def _basic_validate(self, sql: str, dialect: str) -> dict:
        """Basic static validation — catches obvious issues."""
        issues = []
        sql_upper = sql.upper()

        if 'SELECT *' in sql_upper:
            issues.append('WARNING: SELECT * detected — specify columns explicitly')
        if dialect == 'snowflake' and 'NOLOCK' in sql_upper:
            issues.append('ERROR: NOLOCK hint is T-SQL only, not valid in Snowflake')
        if dialect == 'tsql' and 'QUALIFY' in sql_upper:
            issues.append('ERROR: QUALIFY is Snowflake/BigQuery only, not valid T-SQL')
        if sql_upper.count('JOIN') > 8:
            issues.append('WARNING: High join count — verify query performance')

        return {'valid': len([i for i in issues if i.startswith('ERROR')]) == 0, 'issues': issues}


# Usage
generator = MortgageSQLGenerator(api_key='your-key', schema_registry_path='mortgage_schemas.json')

result = generator.generate_sql(
    request="""Compute the 3-month rolling average CPR for each servicer,
               for the last 12 months, excluding loans in foreclosure status.
               Rank servicers by CPR within each reporting month.""",
    tables=['loan_performance', 'loan_master', 'servicer_master'],
    dialect='snowflake'
)

print(result['sql'])
print(f"Tokens used: {result['input_tokens']} in / {result['output_tokens']} out")
```

## Limitations and Risks

### SQL Hallucinations — Common Patterns

| Hallucination Type | Example | Detection |
|---|---|---|
| Non-existent column | `SELECT note_rate_adjusted` (column doesn't exist) | Schema validation |
| Wrong JOIN key | Joins on `loan_id` when FK is `loan_key` | Schema validation |
| Vendor syntax error | Uses `QUALIFY` in T-SQL | Dialect linter |
| Wrong function | `DATEADD('month', 1, date)` in Snowflake (correct: `DATEADD(month, 1, date)`) | Test suite |
| Incorrect aggregation | `AVG(upb)` when `SUM(upb)` is semantically required | Business rule tests |
| Missing NULL handling | Division without `NULLIF` denominator check | Code review |

## Best Practices for AI-Assisted Data Engineering

1. **Schema first**: Always include relevant table schemas in the prompt. AI without schema context generates generic SQL that won't compile.
2. **One complex thing at a time**: Don't ask for a 500-line stored procedure in one shot. Build it in logical sections, validate each.
3. **Test-driven generation**: Write unit tests (dbt tests, pytest) first, then ask AI to generate code that passes them.
4. **Dialect declaration**: Always specify the SQL dialect explicitly. Claude's default SQL leans ANSI; Snowflake and T-SQL have significant differences.
5. **Review every join**: AI most commonly fails on multi-table joins with complex business keys. Always verify join logic manually.
6. **Use AI for the 80%**: Let AI generate boilerplate (column lists, NULL handling, error handling frames). Write the 20% of complex business logic yourself.
7. **Version control AI outputs**: Treat AI-generated code like any other code — it goes through code review and version control before production.
8. **Prompt libraries**: Maintain a library of proven prompts for recurring tasks (index analysis queries, DMV monitoring, CPR calculation). Reusable prompts produce consistent results.

## Interview Q&A

**Q1: How would you use GitHub Copilot to accelerate development of a Snowflake ETL pipeline?**

I use Copilot most effectively by writing detailed comments before the code. For a Snowflake pipeline, I start with a comment block describing the source tables, transformation logic, and target schema. Copilot generates the CTE structure, column mappings, and window functions from that context. Where Copilot helps most: repetitive column alias lists (I write the first 3, Copilot completes the remaining 20), window function frames (it correctly infers ROWS vs RANGE from context), and error handling boilerplate. Where I override Copilot: any multi-table join logic (I verify the keys manually), any conditional aggregation with business-specific rules (WAC, WAM calculations), and any dynamic SQL. The productivity gain is roughly 40-60% on boilerplate-heavy tasks; near zero on novel business logic.

**Q2: What is the most significant risk of using AI code generation for data pipelines in a regulated environment?**

SQL hallucinations that produce silently wrong results rather than errors. A syntax error is caught immediately. A semantically wrong join — for example, AI joining loan performance to a loan master on `loan_id` when the correct key is a composite of `loan_id` and `pool_prefix` — produces results that look plausible but are wrong. In a mortgage context, this could mean CPR calculations based on mismatched loan-pool combinations, affecting hedge ratios or capital allocation decisions. Mitigation requires: (1) schema validation layer that checks all column and table references before execution, (2) result validation against known benchmarks (compare AI-generated CPR to a manually verified reference), (3) mandatory code review for any AI-generated SQL before promotion to production, (4) dbt tests that enforce business rules as invariants.

**Q3: How does Cursor's codebase indexing differ from GitHub Copilot for large data engineering projects?**

Copilot uses the currently open file and a few adjacent files as context — effective for completing a function you're writing but limited for cross-file awareness. Cursor indexes the entire repository and creates embeddings that allow the model to retrieve relevant code from anywhere in the codebase. For a dbt project with 200 models, Cursor can generate a new model that correctly references upstream models by their actual names, follows the exact CTE style used across the project, and matches the test patterns in other models. For a SQL Server project with 500 stored procedures sharing common utility procedures, Cursor can generate a new procedure that calls the right utilities with correct parameter signatures. The practical difference: Cursor generates code that fits your project; Copilot generates code that fits the file.

**Q4: Walk me through building a custom SQL generation tool using the Claude API.**

The key components are: (1) a schema registry — a JSON or database table storing all table definitions (columns, types, relationships, business descriptions); (2) a table detection layer — given a natural language request, identify which tables are relevant (can use keyword matching or a preliminary LLM call); (3) a system prompt builder — injects dialect, coding standards, and relevant schema sections into the system prompt; (4) the generation call — Claude API with the system prompt and user request; (5) static validation — check generated SQL for dialect errors, missing tables, obvious anti-patterns before returning; (6) result logging — store every request, generated SQL, token usage, and validation result for audit and prompt improvement. In production, add a feedback loop: when engineers correct AI-generated SQL, those corrections feed back as few-shot examples in future prompts for similar queries.

**Q5: What is test-driven development with AI and how does it apply to data pipeline code?**

TDD with AI inverts the normal AI workflow: instead of asking AI to write code and then writing tests, you write the tests first and ask AI to write code that passes them. For a Snowflake pipeline: define dbt tests (not_null, accepted_values, relationships, custom SQL tests for business rules like CPR between 0 and 1) before writing the model SQL. Then prompt Claude: "Write a dbt incremental model for CPR calculation that passes these tests: [paste test file]." This approach dramatically reduces hallucinations because the AI has a verifiable target — if the generated SQL fails a test, you iterate with the error message. It also forces test coverage before code exists, which is especially valuable in regulated environments where test documentation is required.

**Q6: How do you handle Snowflake vs T-SQL dialect differences when using AI code generation?**

Explicit dialect declaration in every prompt is the foundation. Beyond that, I maintain a dialect difference cheat sheet that I inject into the system prompt for any non-trivial generation task. Key differences that AI frequently gets wrong: QUALIFY clause (Snowflake only), TOP vs LIMIT (T-SQL vs Snowflake), DATEADD syntax (T-SQL: `DATEADD(month, 1, date)` vs Snowflake: same but case-insensitive), ISNULL vs IFNULL (T-SQL vs Snowflake), window function IGNORE NULLS support, PIVOT syntax, and temporary table handling (T-SQL: `#temp_table`, Snowflake: `CREATE TEMPORARY TABLE`). I also maintain a post-generation linter that scans for T-SQL-specific syntax in Snowflake-targeted code and vice versa.

**Q7: How would you use AI to migrate SSIS packages to Python/Airflow?**

SSIS packages are XML files — Claude can read the XML and understand the data flow. The process: (1) export each SSIS package as XML; (2) prompt Claude with the full XML and ask for a Python equivalent using pandas/SQLAlchemy for transformations and Airflow for orchestration; (3) Claude generates a rough mapping — each Data Flow Task becomes a Python function, each Connection Manager becomes a connection string config, each transformation component (Derived Column, Lookup, Conditional Split) becomes a pandas operation; (4) manually review the generated Python, especially for complex lookups and slowly changing dimension logic; (5) write pytest tests against the original SSIS output to validate the Python migration produces identical results. The AI handles the structural translation well; the manual effort is validating business logic in complex transformations and handling SSIS-specific features (fuzzy lookup, slowly changing dimensions) that have no direct Python equivalent.

**Q8: What are the token cost and latency trade-offs of using Claude API vs GPT-4o for SQL generation at scale?**

For batch SQL generation (e.g., auto-generating dbt model documentation or index recommendations across a schema), cost is material. As of early 2026: Claude Sonnet is cost-effective for high-volume generation with moderate complexity; Claude Opus is best for complex multi-table queries requiring deep reasoning; GPT-4o is competitive for simpler generation tasks. Token costs scale with schema context size — a full mortgage database schema injected into every request can be 2,000-5,000 tokens of input before the user request. Optimization strategies: cache schema embeddings and only inject relevant tables (use a cheap embedding lookup to find relevant tables before the expensive generation call); use Claude Haiku for simple queries (column renaming, format conversions) and Opus only for complex business logic; implement a request cache — identical natural language requests return cached SQL without a new API call.

**Q9: How do you validate AI-generated stored procedures before deploying to a production SQL Server?**

Four-layer validation: (1) Static analysis — run through a SQL linter (SQLFluff with T-SQL dialect) to catch syntax errors, naming violations, and anti-patterns (SELECT *, implicit conversions, non-SARGable predicates); (2) Schema validation — parse the SQL AST to extract all referenced tables and columns, verify them against `information_schema`; (3) Functional testing — deploy to a dev database with a sample of production data, execute with representative parameter values, compare outputs against manually computed expected values; (4) Performance testing — run the execution plan with `SET STATISTICS IO ON; SET STATISTICS TIME ON;`, verify no unexpected table scans, reasonable logical reads. Only after all four layers pass does the procedure go through code review and PR process. I also add a comment block at the top of AI-generated procedures: `-- Generated with AI assistance. Reviewed by: [engineer]. Validated: [date].` for audit trail.

**Q10: A junior engineer on your team is using AI to generate all their SQL without understanding what's produced. How do you handle this?**

This is a code quality and risk management issue, not an AI policy issue. The solution is structured, not prohibitive. First, establish that AI is a tool, not a replacement for understanding — just as using ORMs doesn't excuse not knowing SQL. Second, implement mandatory code review where the reviewer asks the author to explain the logic of any AI-assisted code. If they can't explain the join logic or why a specific window function frame is used, the PR is rejected and they must fix it themselves before resubmitting. Third, add automated safeguards: the static analysis and schema validation pipeline runs on every PR, so hallucinated column names and wrong dialect syntax are caught programmatically. Fourth, create a learning path: pair the junior engineer with a senior reviewer specifically on the complex sections of AI-generated code, making it a teaching moment rather than a blocker. The goal is engineers who can critically evaluate AI output, not engineers who avoid AI or blindly ship it.

## Pro Tips

- **Schema injection over fine-tuning**: For most teams, injecting schema context into prompts is more cost-effective than fine-tuning a code model on your SQL. Fine-tuning is only worth it when you have thousands of company-specific patterns and very high generation volume.
- **Copilot comment-to-code ratio**: The more precise your comments, the better Copilot's output. For T-SQL, a 5-line comment describing parameters, joins, and business logic produces better code than asking Copilot to infer everything.
- **Cursor's @ references**: In Cursor, using `@filename` in your prompt to reference specific files produces dramatically better results than describing the file contents in natural language. Always reference your dbt schema YML files when generating related models.
- **Claude's 200K context window**: For complex stored procedure migration or full-schema analysis, Claude's long context is a genuine competitive advantage. You can paste an entire stored procedure library (50+ procedures) and ask for a dependency graph or refactoring plan.
- **Prompt versioning**: Store your best prompts in a prompts/ directory in your repo, versioned with git. A well-crafted prompt for "generate a Snowflake incremental dbt model" has real reuse value across the team.
- **Two-pass generation**: For complex SQL, use a two-pass approach. Pass 1: ask Claude to design the query structure in pseudocode/comments. Review and correct the logic. Pass 2: ask Claude to implement the SQL from the approved pseudocode. This separates logical correctness (which you verify) from syntax (which Claude handles).
- **AI for SQL review, not just generation**: Paste a complex stored procedure and ask Claude "identify potential performance issues, incorrect NULL handling, or logic errors in this procedure." This use case has fewer hallucination risks than generation and adds significant review throughput.
